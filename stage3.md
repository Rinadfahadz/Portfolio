User Stories & MoSCoW PrioritizationTo ensure the MVP is delivered on time, features are prioritized using the MoSCoW method.Must Have (MVP Core)Profile Management: As a participant, I want to create a profile with my skills and roles so that I can be discovered by teams.Team Creation: As a team lead, I want to create a team and list missing roles so that I can find the right collaborators.Request System: As a user, I want to send join requests to teams so that I can participate in the hackathon.Idea Protection (MVP Version): As a contributor, I want my chat messages timestamped so that there is a verifiable record of my ideas.Should HaveMatching Algorithm: As a user, I want to see "Recommended Teams" based on my skill set.Dashboard: A central view for managing incoming and outgoing requests.Could Have / Won't Have (Post-MVP)GitHub/LinkedIn API integration.Real-time Video calls.AI-powered personality matching.2. System ArchitectureThe application follows a Client-Server Architecture to ensure separation of concerns.Frontend: Built with React.js for a responsive, component-based UI.Backend: Node.js with Express handling RESTful API requests.Database: PostgreSQL (Relational) to manage complex links between users, teams, and requests.Authentication: JWT (JSON Web Tokens) for secure user sessions.3. Database DesignWe are using a Relational Schema to maintain data integrity across the platform.ER Diagram EntitiesUsers: id, username, email, password_hash, primary_role, skills_list[], bio.Teams: id, team_name, description, creator_id (FK), is_full (bool).Requests: id, user_id (FK), team_id (FK), status (pending, accepted, rejected).IdeaLogs (Chat): id, team_id (FK), user_id (FK), message_content, timestamp.4. Sequence Diagram: Team Join FlowThis diagram illustrates the interaction between components when a user requests to join a team.User clicks "Join Request" on Frontend.Frontend sends POST /api/requests with userId and teamId to Backend.Backend validates if the user is already in a team in the Database.Database confirms availability and saves the request as "Pending."Backend returns a 201 Created status to Frontend.Frontend updates the UI to show "Request Sent."5. SCM & QA StrategySource Control Management (Git)Branching Strategy: * main: Production-ready code only.develop: Integration branch for testing.feature/ (e.g., feature/idea-protection): Individual tasks for Rinad and Afnan.Code Reviews: All Pull Requests must be reviewed by at least one other team member before merging into develop.Quality Assurance (Testing)Unit Testing: Using Jest for backend logic (especially the matching algorithm).API Testing: Using Postman to validate status codes and JSON structures.Manual Testing: Weekly "Bug Bashes" where the team tests the UI flow on different browser sizes (Chrome, Safari, Mobile).7. Technical JustificationsWhy PostgreSQL? The relationship between users, their many skills, and their various team applications requires a relational structure to prevent data duplication.Why React? The dashboard requires real-time UI updates (like accepting a request), which React handles efficiently via its state management.Why Timestamped Chat? To solve the "Idea Theft" problem identified in Stage 1, we provide a non-editable log that acts as a simple legal/verifiable trail for participants.
